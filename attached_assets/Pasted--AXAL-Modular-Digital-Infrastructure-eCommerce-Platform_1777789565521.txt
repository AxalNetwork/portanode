# AXAL — Modular Digital Infrastructure eCommerce Platform
## Complete Replit Build Plan (Pre-Sales, Global Shipping)

**Stack constraint:** GitHub Pages (Jekyll) + Cloudflare (Pages, Workers, D1, R2, KV, Turnstile) + Stripe. No traditional servers. No databases outside Cloudflare. No payment processors other than Stripe. Pre-sales only — deposits and reservations, no in-stock fulfillment.

---

## 0. Strategic Frame — Read Before Building

The current axal.io positions Axal as a sustainability services company. The rebrand pivots Axal into a **product-first, configure-to-order eCommerce platform** for modular cargo-container infrastructure. The hero product is the **Digital Infrastructure module** — a plug-and-play modular data center container. Every other module (Energy, Water, Food, Shelter, Waste, Health, Education) is sold as a companion that connects to it through a standard interconnect spec. Customers buy modules à la carte to assemble any business: a research lab, an off-grid clinic, an edge compute node, a vertical farm with on-site compute, a satellite ground station, a disaster-relief deployment.

Because the platform is **pre-sales only**, the eCommerce flow is not "cart → ship today" — it's "configure → quote → reserve with deposit → manufacturing slot → ship globally." Treat every container as a build slot, not inventory.

## Phase 6 — Configurator (the heart of the platform)

40. **Architecture decision.** The configurator is a client-side React or Svelte app embedded in the Jekyll page at `/configure/`. Pick **Svelte + Vite** for bundle size. Build it in `/site/configurator/` and emit a single `bundle.js` + `bundle.css` consumed by the Jekyll page. State management: Svelte stores. No SSR — static site.
41. **Data model.** Define a JSON schema (`/site/_data/catalog.json`) describing every module, every option, every constraint. Each module has: `id`, `name`, `basePrice`, `currency`, `dimensions`, `weight`, `power`, `options[]` (each option has `id`, `name`, `priceDelta`, `weightDelta`, `powerDelta`, `requires[]`, `excludes[]`, `category`), `interconnects[]` (which modules it can mate with), `leadTimeWeeks`, `regions[]` (where it can ship).
42. **Configurator UI — left rail.** Module picker. User starts with one module (defaults to AXAL Core if entering from homepage CTA, or the chosen module if entering from a product page). User can add additional modules to build a Stack.
43. **Configurator UI — center stage.** A live 3D or isometric 2D rendering of the configured stack. Use **three.js** for 3D or **Konva** for 2D isometric. 3D is sexier and worth the bundle cost on a $50k+ purchase decision. Show modules snapping together as user adds them.
44. **Configurator UI — right rail.** The spec ledger. Live-updating list of: current configuration, line items with prices, running subtotal, total power draw, total weight, total footprint, estimated lead time (max of all module lead times), shipping region selector, estimated shipping cost, grand total.
45. **Per-module option editor.** Click a module in the canvas → modal opens showing all options for that module grouped by category (e.g., for AXAL Core: Compute Tier, Storage Tier, Network, Cooling, Power Input, Climate Hardening, Security Package, Monitoring & AI, Connectivity Add-ons). Each option shows price delta in real time. Constraint engine prevents invalid combinations (e.g., can't pick "Liquid Cooling" without "High-Density Compute").
46. **Constraint engine.** Implement `requires` and `excludes` rules from the catalog JSON. When a user picks an option, automatically check/uncheck dependencies and flag conflicts with inline tooltips.
47. **Interconnect validation.** When a Stack has multiple modules, validate that the combined power draw doesn't exceed what the chosen Volt module can supply, that water demands match Flow capacity, that footprint fits the customer's claimed site dimensions (optional input).
48. **Save / share / resume.** Every configuration generates a unique short ID (e.g., `cfg_a4Bc9X`) stored in Cloudflare KV. URL pattern `/configure/?c=cfg_a4Bc9X`. Customer can email the link to themselves or their team, return later, or share with a colleague. Shared configs are read-only until claimed by signing in.
49. **Print/PDF spec sheet.** Button on the configurator and quote pages: "Download spec sheet." A Cloudflare Worker generates a branded PDF of the configuration using a serverless PDF library or by rendering an HTML print stylesheet to PDF via headless service (Browserless on Workers). Include AXAL letterhead, full BOM, total price, validity (30 days), and a quote ID.
50. **Mobile UX.** The configurator must work on mobile. Below 768px, collapse the 3-column layout to a tabbed view: Modules / Canvas / Specs. The 3D canvas degrades to a static isometric SVG on low-end devices.

## Phase 7 — Quote, Pre-Order & Stripe Integration

51. **Quote vs. Pre-Order distinction.** A *quote* is a saved configuration with an estimated price and 30-day price lock — no payment. A *pre-order* is a quote with a deposit paid through Stripe, which reserves a manufacturing slot.
52. **Quote flow.** From the configurator: "Get formal quote." Customer enters company name, contact name, email, phone, country, intended deployment site (optional), use case (dropdown). Quote is generated, emailed, stored in D1, and shown at `/quote/{quote_id}/` (signed URL, no auth required, valid 30 days).
53. **Stripe products and prices.** Do **not** create one Stripe product per module — create a small set of Stripe products: "Module Deposit," "Stack Deposit," "Customization Fee," "Expedite Fee," "Shipping," and use dynamic prices (`price_data` in Checkout Sessions) calculated per-quote by the Worker. This avoids managing thousands of Stripe SKUs.
54. **Deposit policy.** Standard deposit = 20% of total. Show this clearly. The remaining 80% is invoiced at manufacturing slot confirmation (manual process, AXAL ops triggers the second invoice via Stripe Invoices API).
55. **Stripe Checkout Sessions.** When a customer accepts a quote and pays the deposit, a Worker creates a Checkout Session in `payment` mode with: line items (deposit + customization + shipping), customer email, success URL → `/account/orders/{order_id}/?success=1`, cancel URL → quote page, billing address required, shipping address required, payment methods: cards + ACH Direct Debit (US) + SEPA + Bank transfer. Enable Stripe Tax for global VAT/GST.
56. **3D Secure & SCA.** Stripe handles automatically — confirm enabled on account.
57. **Webhooks.** Cloudflare Worker endpoint `/api/stripe/webhook` listens for `checkout.session.completed`, `payment_intent.succeeded`, `payment_intent.payment_failed`, `invoice.paid`, `invoice.payment_failed`, `charge.refunded`, `charge.dispute.created`. Verify the signature using `STRIPE_WEBHOOK_SECRET`. On `checkout.session.completed`: persist the order in D1, set status to `RESERVED`, send confirmation email, fire CRM update.
58. **Idempotency.** Use Stripe's `Idempotency-Key` header on every write call to prevent double-charges on retries. Generate keys deterministically from the quote ID and action.
59. **Refund flow.** Per the Refund & Cancellation Policy: full refund within 14 days of deposit, 50% within 30 days, 0% after manufacturing slot confirmed. Build an admin-triggered refund endpoint that calls Stripe Refunds API and updates D1.
60. **Multi-currency.** Stripe supports presentment currency. Detect customer country (Cloudflare provides `cf.country` in Worker request), convert prices using a daily-cached FX rate from a public API (ECB or `exchangerate.host`) stored in KV with a 24h TTL. Default to USD; show local currency where supported by Stripe (EUR, GBP, CHF, CAD, AUD, JPY, etc.).
61. **Tax calculation.** Enable Stripe Tax on the account. For B2B sales with valid VAT IDs, support reverse charge (collect VAT ID at checkout, validate via VIES for EU, store in customer record).

## Phase 8 — Cloudflare Workers Backend (`/workers`)

62. **Project setup.** Use **Hono** as the Worker framework — small, fast, TypeScript-first, plays well with Workers and D1. Structure: `/workers/src/{routes,services,db,email,stripe,middleware}/`.
63. **Public API endpoints.** `POST /api/configurations` (save a config, returns short ID), `GET /api/configurations/:id` (resume), `POST /api/quotes` (create quote from config + customer info), `GET /api/quotes/:id` (view quote), `POST /api/quotes/:id/checkout` (create Stripe Checkout Session), `POST /api/quotes/:id/pdf` (generate quote PDF), `POST /api/contact` (general contact form), `POST /api/leasing` (leasing inquiry form), `POST /api/spec-download` (gate the spec sheet behind an email).
64. **Auth.** Customer accounts use **email-link magic auth** — no passwords. Worker generates a one-time JWT, emails it via Resend, customer clicks, JWT is exchanged for a session cookie (HttpOnly, Secure, SameSite=Lax, 30-day TTL). No social logins (B2B audience doesn't need them).
65. **Customer portal endpoints.** `GET /api/account/orders` (list orders), `GET /api/account/orders/:id` (order detail with timeline), `GET /api/account/quotes` (list quotes), `POST /api/account/orders/:id/notes` (customer notes / questions), `GET /api/account/invoices/:id` (Stripe invoice PDF passthrough).
66. **Admin endpoints (protected by `ADMIN_API_TOKEN`).** `GET /admin/orders`, `PATCH /admin/orders/:id` (update status), `POST /admin/orders/:id/invoice-balance` (trigger 80% invoice), `POST /admin/refunds`, `GET /admin/quotes`, `GET /admin/leads`, `GET /admin/dashboard` (counts, MRR-equivalent of deposit pipeline).
67. **D1 schema.** Tables: `customers` (id, email, company, country, vat_id, created_at), `configurations` (id, payload_json, created_at, claimed_by_customer_id), `quotes` (id, configuration_id, customer_id, total_cents, currency, status, expires_at, pdf_url), `orders` (id, quote_id, customer_id, deposit_amount_cents, balance_amount_cents, status, stripe_checkout_session_id, stripe_payment_intent_id, manufacturing_slot, ship_eta, tracking, created_at, updated_at), `events` (id, type, entity_id, payload_json, created_at — append-only event log for audit), `leads` (id, source, payload_json, status, created_at), `email_subscriptions` (id, email, list, double_opted_in, created_at).
68. **Migrations.** Use `wrangler d1 migrations create` to manage schema. Every change is a migration file in `/workers/migrations/`. Never edit D1 by hand in production.
69. **Rate limiting.** Use Cloudflare's rate-limiting rules + Turnstile on every public POST endpoint to prevent spam quote generation. Aggressive on `/api/contact` (3/min/IP), generous on `/api/configurations` (60/min/IP).
70. **Logging & observability.** Use Cloudflare Workers Logs. Add structured JSON logging with request ID, customer ID (when known), endpoint, status, latency. Pipe to Logpush for retention if budget allows; otherwise rely on Tail for live debugging.
71. **Error handling.** All endpoints return a uniform error envelope `{error: {code, message, details}}` with appropriate HTTP status. Never leak stack traces. Log full error server-side, return user-safe message client-side.

## Phase 9 — Lightweight CRM

72. **Decide build vs. buy.** A full CRM (HubSpot, Pipedrive) is overkill at launch. Build a minimal CRM inside the admin panel powered by D1. Migrate to a real CRM later via export. The admin panel is a separate Jekyll-rendered HTML page protected by basic auth at the Cloudflare edge, with a JS app calling `/admin/*` endpoints.
73. **Lead capture.** Every form on the marketing site (contact, leasing, spec download, newsletter, demo request) writes to the `leads` table with a `source` field. Admin sees a unified inbox.
74. **Pipeline stages.** Define lead → qualified → quote sent → deposit paid → manufacturing → shipped → delivered → closed. Each order moves through this pipeline. Build a Kanban view in the admin panel (drag-and-drop persists status to D1).
75. **Customer view.** For each customer: profile, all configurations, all quotes, all orders, all communications, manual notes (free-text, timestamped). Search by email or company.
76. **Tasks & reminders.** Simple task list per customer/order. "Follow up with Acme on quote Q-1024 in 7 days." Email digest to ops@ each morning with overdue tasks (Worker cron trigger).
77. **Email integration.** Outbound emails (quote, order confirmation, shipping update, balance invoice) sent through Resend; every send writes a row to `events` with `type='email_sent'`. Inbound replies are not handled in v1 — emails go to a real shared inbox (Google Workspace), and ops manually log notes in the admin panel.
78. **Export.** "Export all" button → ZIP of CSVs (customers, quotes, orders, leads, events). Run as a cron-triggered Worker that writes to R2 weekly so a snapshot is always recent.

## Phase 13 — Compliance, Risk & Legal

96. **Export controls.** Modular data center containers can hit dual-use export controls (EU 2021/821, US EAR). Build a country gate: when shipping address is in an embargoed/sanctioned country (OFAC, EU, UK, UN lists), block checkout with a polite explanation and capture the lead for manual review. Maintain the country list in KV, refreshable monthly by ops.
97. **KYB on large orders.** Any order over a configurable threshold (e.g., $250k) triggers a manual KYB step before the manufacturing slot is confirmed. Customer is notified that their deposit is held in escrow status until KYB clears. Use a third-party (Sumsub, Onfido, Persona) — link from admin panel.
98. **Sanctions screening.** On every customer creation, screen the company name and primary contact against OFAC SDN list via a free or low-cost API (OpenSanctions, ComplyAdvantage). Flag matches for ops review.
99. **GDPR / FADP.** Cookie banner with granular consent (necessary, analytics, marketing). Use a privacy-respecting analytics tool (Plausible, Cloudflare Web Analytics) that doesn't require consent for basic pageviews. Provide a `/legal/privacy/` page with full disclosures and a `/account/privacy/` self-service for data export and deletion (Right to Access, Right to Erasure). Implement these as Worker endpoints.
100. **PCI scope minimization.** Never touch card data. Stripe Checkout (hosted) keeps you in SAQ-A scope. Do not embed Stripe Elements unless you accept the additional compliance burden — for v1, use Checkout.
101. **Terms of Sale.** Specify Geneva, Switzerland as governing law and venue, with an arbitration clause (ICC or Swiss Chambers). Cap liability at the price of the goods. Disclaim consequential damages. Have a Swiss attorney review before launch — this is non-negotiable for cross-border B2B at these price points.

## Phase 14 — Analytics & Conversion Instrumentation

102. Install **Cloudflare Web Analytics** (free, privacy-respecting, no cookie banner needed for it). Use it as the source of truth for pageviews and core web vitals.
103. Add **Plausible** or self-hosted **Umami** for funnel analysis. ## Phase 16 — Ops Runbook & Day-Two

114. **Runbook for receiving an order.** Step-by-step doc for ops: deposit notification → KYB if applicable → manufacturing slot assignment → customer email → balance invoice in Stripe → balance paid → ship → tracking entry → delivery confirmation. Live in the repo as `/docs/runbook.md`.
115. **Incident response.** On-call rotation, alerting via Cloudflare Workers monitoring + Stripe webhook failures + DNS uptime monitoring (UptimeRobot or Better Stack). Escalation tree.
116. **Backups.** D1 export weekly to R2, retained 13 months. KV is cache; loss is acceptable. R2 has built-in durability.
117. **Disaster recovery.** Document the steps to restore from D1 export, redeploy Workers, restore secrets, and rotate Stripe keys if compromised. Test quarterly.
118. **Pricing review cadence.** Module prices are in `_data/catalog.json` — version-controlled. Quarterly review with finance. Promotions and regional adjustments via separate `_data/promotions.json` consumed by the configurator.
119. **Manufacturing capacity.** The system needs to know slot capacity to prevent over-selling. Add a `manufacturing_capacity` table in D1 with month-by-month slots per module type. Configurator reads it (cached) to show realistic lead times. When capacity for a month sells out, the Worker auto-shifts the displayed lead time for new quotes.

## Phase 17 — Pre-Launch QA

120. End-to-end test happy path: configure → quote → deposit → confirmation email → admin sees order → balance invoice → status updates → delivered.
121. End-to-end test edge paths: expired quote, abandoned checkout, refund, sanctions hit, KYB hold, payment failure, currency edge cases (small currencies, no decimals like JPY), large orders.

124. Pen test the public surface: SQL injection (D1 prepared statements only), XSS (Jekyll auto-escapes; configurator must sanitize URL params), CSRF (state-changing POSTs require the auth cookie + a CSRF token), webhook signature verification, JWT validation, rate limiting.
125. Privacy test: trigger a GDPR data-export from a test account, verify all data is included, verify deletion actually removes records.
126. Stripe test mode dry runs: every webhook event, every refund scenario, every dispute scenario.
